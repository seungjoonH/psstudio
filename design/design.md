# PS Study Product Design

> 구현 전에 제품 범위, 사용자 흐름, 정책, 확정된 결정을 정리하는 기획서입니다.

## 1. 제품 한 줄 정의

psstudio 는 알고리즘 문제 풀이를 그룹 단위로 운영하면서, 과제, 제출 현황, 코드 리뷰, 댓글, 알림, AI 분석을 한 흐름 안에서 관리하는 스터디 플랫폼입니다.

## 1.1 용어

- 과제는 특정 문제 하나에 대해 그룹원이 코드를 제출하고 피드백을 주고받는 작업 단위입니다.
- 기존 대화의 `문제 세션`과 `풀이 세션`은 문서에서 `과제`로 부릅니다.
- 과제 하나는 반드시 문제 하나만 가집니다.
- 문제 묶음, 대회, 숙제 세트처럼 여러 문제를 묶는 기능은 별도 상위 개념이 필요할 때 추가합니다.

## 1.2 공개 랜딩

- `/landing`은 인증 없이 접근 가능한 제품 소개 페이지입니다. 본문은 로그인 화면과 동일한 `AppShell`(사이드바·상단 타이틀 바)과 카드형 서피스로 구성합니다. 짧은 설명 문구와 예시가 채워진 정적 UI 더미(홈 학습 현황 보드·그룹 캘린더·과제 목록 카드·diff·리뷰 댓글·AI 피드백·알림 목록 등)는 실제 화면과 동일한 CSS 모듈을 쓰고 데이터만 고정합니다. `/public/landing/hero-mesh.svg` 장식으로 시각 비중을 둡니다. `IntersectionObserver`로 뷰포트 진입 시 블록별 페이드인·살짝 상승(`ScrollReveal`)하며, `prefers-reduced-motion`이면 전환을 끕니다. 검색·소셜 공유를 위해 서버에서 `title`·`description`·canonical·Open Graph·Twitter 카드·JSON-LD(WebPage·WebSite·SoftwareApplication) 메타데이터를 구성합니다. 본문 카피는 i18n(`landing.*`)으로 한국어·영어를 제공합니다.

## 3. 사용자별 목적

### 그룹장

- 스터디 그룹을 만들고 구성원을 초대합니다.
- 그룹을 삭제하거나 그룹장 권한을 다른 그룹원에게 넘깁니다.
- 그룹원을 그룹 관리자로 승격하거나 그룹 관리자를 그룹원으로 변경합니다.
- 특정 문제에 대한 과제를 생성합니다.
- 그룹 규칙 기본값을 따르는 과제를 만들고, 필요하면 deadline 시각과 지각 허용 여부만 과제별로 override 합니다.
- 그룹 캘린더에서 구성원의 제출 여부를 확인합니다.
- 공지와 커뮤니티 글을 통해 스터디 운영 정보를 공유합니다.

### 그룹 관리자

- 그룹장의 위임을 받아 그룹원 초대와 강퇴를 수행합니다.
- 과제를 생성하고 수정합니다.
- 과제를 삭제합니다.
- 가입 신청을 승인하거나 거절합니다.
- 그룹 캘린더에서 구성원의 제출 여부를 확인합니다.
- 부적절한 댓글을 삭제합니다.
- 그룹 삭제를 제외한 그룹 운영 권한은 그룹장과 동일합니다.

### 그룹원

- 그룹 캘린더에서 오늘 풀어야 할 문제와 제출 상태를 확인합니다.
- 문제 링크를 열어 문제를 풉니다.
- 과제에 본인의 풀이 코드를 텍스트로 붙여 넣습니다.
- AI가 작성한 코드 리뷰 댓글을 확인하고 답글을 답니다.
- 다른 사람의 풀이에 댓글과 코드 리뷰를 남깁니다.
- 본인 제출물에 댓글, 리뷰, 피드백 공개 등이 생기면 알림을 받습니다.

## 4.1 로컬 인프라 구성

로컬 개발은 Docker Compose를 기본 실행 방식으로 사용합니다.

논리 서비스는 다음 5개로 둡니다.

- `fe`: Next.js 프론트엔드.
- `be`: NestJS API 서버.
- `worker`: AI 분석, 문제 분석, deadline 임박 알림, 백그라운드 알림 처리.
- `supabase`: 로컬 개발용 Supabase Postgres 호환 DB.
- `redis`: 큐, 작업 상태, 알림 카운트, 락.

Supabase는 운영 또는 통합 개발 환경에서 Supabase Cloud나 Supabase CLI 로컬 스택으로 교체할 수 있습니다. 이 경우 앱 컨테이너는 `DATABASE_URL`만 바꿔 연결합니다.

### 4.1.1 백엔드 환경변수(메일·LLM)

로컬에서는 워크스페이스 루트가 아니라 `be/.env.local`만 사용합니다. 값은 저장소에 넣지 않습니다.

- 이메일 발송(Resend): `RESEND_API_KEY`, `EMAIL_FROM_ADDRESS`.
- LLM 호출용 provider/키/모델명: `LLM_PROVIDER`(`openrouter|requesty`), `OPEN_ROUTER_API_KEY`, `REQUESTY_API_KEY`, `REQUESTY_BASE_URL`, `LLM_MODEL_PROBLEM_ANALYZE`(집단 분석 등), `LLM_MODEL_ASSIGNMENT_AUTOFILL`(과제 생성「AI 자동 채우기」전용), `LLM_MODEL_SUBMISSION_REVIEW` 등 BE 부팅 시 필수로 읽는 키는 `design/api-spec.md` §1과 동일하게 유지합니다.

## 4.2 전역 UI 인프라

프론트엔드는 라이트/다크 모드와 한국어/영어 전환을 전역 인프라로 제공합니다.

- 기본 테마는 시스템 설정을 따릅니다.
- 사용자가 테마를 직접 선택하면 브라우저 localStorage에 저장합니다.
- 테마 값은 `system`, `light`, `dark`를 지원합니다.
- 실제 색상은 CSS 변수 기반 토큰으로 관리합니다.
- 기본 언어는 한국어입니다.
- fallback 언어는 영어입니다.
- 언어 전환은 MVP에서 라우팅을 바꾸지 않고 클라이언트 상태와 localStorage로 처리합니다.
- 번역 리소스는 한국어와 영어의 키 구조가 항상 같아야 합니다.
- 날짜와 시간은 KST 기준 값을 locale에 맞게 표시합니다.
- 코드, 문제 플랫폼 이름, 알고리즘 태그, 프로그래밍 언어 이름은 번역하지 않는 용어로 취급합니다.

## 5. 핵심 흐름

### 5.1 그룹 생성과 구성원 관리

1. 사용자가 그룹을 생성합니다.
2. 그룹을 만든 사용자는 기본적으로 그룹장이 됩니다.
3. 그룹은 다음 정보를 가집니다.
   - 그룹명(필수, 최대 20자).
   - 그룹 설명(선택, plain text, 최대 500자).
   - 최대 인원수(필수, 최소 2명, 기본 10명, 시스템 한도 50명).
   - 그룹 코드(필수, 영구 1개, 8자, 영문 대소문자 + 숫자, 대소문자 구분).
   - 가입 방식 허용 토글 4개(코드/링크/가입 신청/이메일 초대).
   - 그룹 규칙(아래 5.1.1).
4. 그룹 프로필 이미지와 그룹 배너 이미지는 MVP에서 제공하지 않습니다.
5. 그룹장 또는 그룹 관리자는 그룹명, 그룹 설명, 최대 인원수, 가입 방식 토글, 그룹 규칙을 수정할 수 있습니다.
6. 최대 인원수는 그룹 생성 후에도 수정 가능하지만 현재 멤버 수보다 작게 줄일 수 없습니다.
7. 가입 시도 시 현재 멤버 수가 최대 인원수에 도달했으면 가입을 거부합니다.
8. 그룹장만 그룹 코드를 갱신할 수 있습니다. 그룹 코드 갱신은 새 코드 1개를 발급하고 이전 코드 및 이전 코드로 만들어진 초대 링크를 즉시 무효화합니다. 그룹 관리자에게는 갱신 버튼을 노출하지 않습니다.
9. 무효화된 코드 또는 링크로 접근하는 시도는 `404 NOT_FOUND` 응답으로 처리합니다.
10. 그룹 코드 자체는 그룹원 모두에게 노출합니다. 그룹원이 코드를 다른 사람에게 공유해 새 멤버를 들이는 것을 막지 않습니다(가입 토글이 켜져 있을 때).
10. 그룹은 공개/비공개 구분이 없습니다. 비밀번호 보호, 공개 검색 같은 개념도 두지 않습니다. 그룹에 들어오는 모든 경로(코드 입력, 초대 링크, 이메일 초대)는 동일한 그룹 코드를 신뢰 토큰으로 사용합니다.
11. 그룹 코드 또는 초대 링크를 가진 사용자는 누구나 그룹에 합류할 수 있습니다. 추가 인증은 없습니다.
12. 사용자는 그룹 코드를 입력해서 그룹에 가입할 수 있습니다(가입 방식 토글의 `코드`가 켜져 있을 때).
13. 사용자는 초대 링크로 그룹에 가입할 수 있습니다(가입 방식 토글의 `링크`가 켜져 있을 때).
14. 사용자는 그룹 가입 신청을 보낼 수 있습니다(가입 방식 토글의 `가입 신청`이 켜져 있을 때).
15. 가입 신청에는 자기소개나 신청 메시지를 받지 않습니다.
16. 그룹장 또는 그룹 관리자는 가입 신청을 승인하거나 거절할 수 있습니다. 정원이 가득 찬 시점에서도 가입 신청 접수 자체는 허용하며, 승인 시점에 정원이 가득 차 있으면 승인을 거부합니다.
17. 그룹장 또는 그룹 관리자는 이메일 주소로 초대를 발송할 수 있습니다(가입 방식 토글의 `이메일 초대`가 켜져 있을 때).
18. 이메일 초대 흐름.
    - 발송 시 1회용 초대 토큰을 만들고 이메일에 그룹 코드와 토큰을 담은 가입 링크를 포함합니다.
    - 토큰 TTL은 7일이며, 한 번 사용되면 만료됩니다.
    - 이메일 수신자가 미가입자이면 가입 링크가 OAuth 로그인 페이지로 안내합니다. 로그인 또는 가입이 끝나면 자동으로 그룹에 합류합니다.
    - 이메일 수신자가 이미 가입자이면 로그인 후 자동으로 그룹에 합류합니다.
    - abuse 방지를 위해 그룹당 1시간 내 최대 50건 발송으로 제한합니다. 초과 시 `RATE_LIMITED`.
    - 같은 이메일에 활성 PENDING 토큰이 이미 있으면 기존 토큰을 만료시키고 새 토큰을 발급합니다(과한 중복 발송 방지).
19. 가입 방식 토글이 모두 꺼져 있으면 어떠한 신규 멤버도 추가되지 않습니다. 그룹장이 토글을 다시 켤 때까지 동결됩니다.
20. 한 가지 토글만 꺼져 있는 경우 다른 토글로 들어오는 가입은 그대로 허용됩니다.
21. `request` 토글이 꺼지더라도 이미 쌓인 `PENDING` 가입 신청은 자동으로 거절되지 않고 그대로 유지됩니다. 토글이 꺼져 있는 동안 그룹장/그룹 관리자는 `PENDING`에 대해 거절(`REJECTED`)만 가능합니다(승인은 막힘). 토글이 다시 켜지면 승인·거절 모두 가능합니다.
22. 정원 검사는 멤버 추가 트랜잭션 안에서 unique 제약과 카운트 비교로 처리합니다(별도 row lock은 두지 않음). 동시 요청 race는 50명 한도 안에서 충분히 작은 빈도이며, 실패 시 `CONFLICT GROUP_FULL`로 응답합니다.
21. 구성원은 그룹에 가입하고 역할을 부여받습니다.
22. 그룹장은 그룹원별 권한을 관리합니다.

#### 5.1.1 그룹 규칙

그룹 규칙은 그룹 생성 시점에 모두 기본값으로 채워집니다. 그룹장 또는 그룹 관리자가 언제든 수정할 수 있습니다.

- `useDeadline`: 마감일 기능 사용 여부. 기본값 `true`. `false`이면 과제에 마감일이 없고 지각 개념도 없습니다.
- `defaultDeadlineTime`: 마감일 사용 시 새 과제의 기본 마감 시각(KST). 기본값 `23:59`.
- `allowLateSubmission`: 마감 이후 제출 허용 여부. 기본값 `true`. `useDeadline`이 `false`이면 의미가 없습니다.
- `useAiFeedback`: AI 피드백(AI 코드 리뷰) 사용 여부. 기본값 `true`. `false`이면 AI는 코드 리뷰를 작성하지 않습니다(자세한 동작은 5.4 참조).
- `allowEditAfterSubmit`: 제출 후 코드 수정 허용 여부. 기본값 `true`. `false`이면 제출 코드의 새 버전 생성이 차단됩니다(메타데이터 수정은 항상 허용).
- `assignmentCreatorRoles`: 과제를 생성할 수 있는 역할 범위. `OWNER_ONLY` 또는 `OWNER_AND_MANAGER`. 기본값 `OWNER_AND_MANAGER`.

피드백 공개 범위는 그룹 규칙에 따로 두지 않습니다. 5.6의 단순화된 정책을 따릅니다.

#### 5.1.2 그룹 진입 경로

그룹 진입 화면은 1개로 통일합니다. 사용자는 그룹 코드를 입력해서 들어오거나, 초대 링크 또는 이메일 초대 링크로 들어옵니다. 모든 경로는 같은 백엔드 엔드포인트 1개를 호출하고, 가입 방식 토글에 따라 허용 또는 거부됩니다.

#### 5.1.3 그룹 탭 라우팅과 그룹 둘러보기

기존의 “내가 속한 그룹 카드 그리드” 화면(`/groups`)은 폐기했습니다. 그룹 탭은 다음 동작만 갖습니다.

- `/groups`는 서버 라우트이며 직접 화면을 그리지 않습니다. 서버에서 가입한 그룹 목록을 조회한 뒤 다음 규칙으로 즉시 `redirect`합니다.
  - 가입한 그룹이 0개이면 `/groups/explore`(그룹 둘러보기)로 이동합니다.
  - 가입한 그룹이 1개 이상이면 쿠키 `psstudio_last_group`(`SameSite=Lax`, max-age 약 400일)에 저장된 마지막 접근 그룹 ID로 이동합니다. 쿠키가 없거나 더 이상 멤버가 아닌 그룹이면 첫 번째 그룹으로 이동합니다.
- 그룹 상세(`/groups/:id`) 진입 시 클라이언트가 `psstudio_last_group` 쿠키와 `localStorage("psstudio:lastGroupId")`를 함께 갱신합니다.
- 그룹 상세 헤더 우측에 “그룹 추가”(모달 오픈)와 “그룹 둘러보기”(`/groups/explore` 이동) 두 버튼을 둡니다. 두 버튼은 같은 공용 `Button` 컴포넌트로 크기를 맞춥니다.
- “그룹 추가” 모달(`AddGroupModal`)은 두 가지 진입을 한 화면에 묶습니다.
  - 새 그룹 만들기(`/groups/new` 이동).
  - 초대 코드로 가입(8자 코드 입력 폼, 기존 `/join-by-code`의 server action을 그대로 재사용).
- “그룹 둘러보기”(`/groups/explore`)는 가입한 그룹이 없거나 다른 그룹을 더 만들고 싶을 때 들어오는 단일 진입점입니다. “새 그룹 만들기”와 “초대 코드로 가입” 두 카드를 같은 화면에 노출합니다. 외부 그룹 검색은 두지 않습니다.
- 그룹 카드 그리드 화면을 다시 만들지 않습니다(중복 화면 금지).

그룹 역할은 다음 3단계로 시작합니다.

- 그룹장.
- 그룹 관리자.
- 그룹원.

그룹장과 그룹 관리자 권한의 핵심 차이는 다음과 같습니다.

- 그룹장은 항상 1명입니다.
- 그룹 관리자는 여러 명일 수 있습니다.
- 그룹장은 그룹 삭제 권한을 가집니다.
- 그룹 삭제 시 그룹명을 입력하고 `삭제된 그룹은 복구할 수 없습니다` 경고 문구를 보여줍니다.
- 그룹장은 다른 그룹원에게 그룹장 권한을 넘길 수 있습니다.
- 그룹장 권한을 넘길 때 새 그룹장의 수락은 필요하지 않습니다.
- 그룹 삭제를 제외한 그룹 운영 권한은 그룹장과 그룹 관리자가 동일합니다.
- 그룹장과 그룹 관리자는 그룹원을 그룹 관리자로 승격하거나 그룹 관리자를 그룹원으로 변경할 수 있습니다.

### 5.2 과제 생성

1. 그룹장이 그룹 캘린더에서 날짜를 선택합니다.
2. 그룹장 또는 그룹 관리자가 과제를 추가합니다.
3. 과제에는 반드시 문제 링크가 포함됩니다.
4. 문제 링크는 외부 URL만 허용합니다.
5. 문제 링크는 지원 플랫폼별 파싱으로 유효성을 검증합니다.
6. 지원 플랫폼 태그는 예를 들어 `BOJ`, `Programmers`, `LeetCode`처럼 저장합니다.
7. MVP 지원 플랫폼은 `BOJ`, `Programmers`, `LeetCode`로 시작합니다.
8. 지원 플랫폼으로 확정되지 않는 외부 URL은 AI가 플랫폼과 문제 정보를 판단합니다.
9. 같은 문제 링크를 여러 과제에 등록할 수 있으며 중복 경고는 표시하지 않습니다.
10. 과제 제목은 문제 분석 결과 제목을 기본값으로 사용합니다.
11. 그룹장 또는 그룹 관리자는 과제 제목을 수정할 수 있습니다.
12. 과제에는 간단한 설명 또는 메모 필드를 둡니다.
13. 그룹장 또는 그룹 관리자는 deadline과 AI 분석 정책을 설정합니다.
14. 과제 설정은 생성 후 언제든지 수정할 수 있습니다.
15. 별도의 과제 관리자 역할은 두지 않고 그룹장과 그룹 관리자가 과제를 관리합니다.
16. 그룹장 또는 그룹 관리자는 과제를 삭제할 수 있습니다.
17. 생성된 과제는 그룹 캘린더와 과제 목록에 표시됩니다.
18. deadline은 날짜와 시각까지 설정하며 기준 시간대는 KST입니다.
19. deadline 기본값은 `23:59 KST`이고 과제별로 커스텀할 수 있습니다.
19-1. 과제 생성 폼에서는 `마감 시간`과 `과제 수행 주기`(기간: 1/3/7/14일 또는 요일: 일~토) 중 하나를 선택해 deadline을 자동 계산해 채우되, 사용자가 `마감일`(`datetime-local`)을 직접 수정할 수 있습니다. 직접 수정 시 현재 선택된 모드(기간/요일)에 맞는 값이 있으면 자동으로 선택되고, 기간 모드에서 14일 초과 또는 preset(1/3/7/14일) 불일치인 값, 요일 모드에서 8일 이상 미래 값은 해당 선택을 해제합니다.
19-2. 과제 생성 폼에는 `알고리즘` 입력을 둡니다. 알고리즘은 미리 정의된 키워드 셋으로 관리하며, 사용자가 `BFS, DFS`처럼 쉼표 입력하면 동의어를 정규화해 미리 정의된 키워드로 치환합니다. 생성 시 선택된 알고리즘 키워드는 과제 메타데이터 `algorithms`에 저장됩니다.
19-3. 과제 생성 폼의 `AI 자동 채우기` 버튼은 문제 URL을 기준으로 백엔드 AI(OpenRouter 또는 Requesty) 호출 결과를 사용해 힌트·알고리즘 초안을 자동 반영하고, **제목은 `Programmers`·`BOJ`·`LeetCode`에서 문제 페이지 HTML의 공식 명칭(`data-lesson-title`, `#problem_title`, `og:title` 등)을 우선 추출**한다. 공식 제목 추출에 실패할 때만 AI가 제안한 `title`을 쓴다. 그 외 플랫폼(`Other`)은 제목도 AI가 채운다. 서버는 AI 호출 전에 문제 페이지 본문 유효성을 검사합니다. `입력`/`출력` 키워드 쌍 또는 코드 에디터 블록(`code-editor`, `codehilite`, `rouge-code`)이 확인되지 않으면 AI 호출을 생략하고 오류를 반환합니다. 난이도는 플랫폼 정책을 따릅니다. `Programmers`와 `BOJ`는 AI 추정값을 쓰지 않고 문제 페이지 메타데이터에서만 확정값을 반영하며, 메타 추출 실패 시 난이도를 비워 둡니다. `LeetCode`와 기타 플랫폼은 AI 응답을 `Easy/Medium/Hard` 등 규칙으로 정규화해 반영합니다. AI는 알고리즘을 최소 1개 이상 반환해야 하며, 서버가 제공한 허용 목록 안에서만 선택해야 합니다. 미반환 또는 허용 목록 불일치면 자동 채우기를 실패 처리합니다. 선택한 provider의 API 키(`OPEN_ROUTER_API_KEY` 또는 `REQUESTY_API_KEY`)가 없거나 호출이 실패하면 폼에서 에러 토스트를 표시하고, 사용자는 저장 전에 값을 직접 수정할 수 있습니다. 사용 모델은 `LLM_MODEL_ASSIGNMENT_AUTOFILL` 환경변수로 지정한다(집단 분석 등 다른 용도는 `LLM_MODEL_PROBLEM_ANALYZE`).
19-4. 과제 힌트와 알고리즘은 공개 속성(`hintHiddenUntilSubmit`, `algorithmsHiddenUntilSubmit`)을 가집니다. 기본값은 `true`이며, 이 상태에서는 과제 생성자를 포함한 모든 그룹원이 기본 화면에서 볼 수 없습니다. 사용자가 `힌트 보기`/`알고리즘 보기`를 누르면 현재 세션에서만 일시적으로 확인할 수 있고, 본인 코드 제출이 1회 이상 존재하면 해당 사용자에게는 영구 공개됩니다. 제출이 다시 0건이 되면 공개 상태도 즉시 해제됩니다.
19-5. 그룹 캘린더 날짜 셀의 `+` 버튼으로 과제 생성 페이지에 진입하면, 선택한 날짜가 과제 생성 폼 `마감일` 기본값으로 자동 채워집니다(시간은 그룹 기본 마감 시각 사용).
20. deadline 이후 제출 허용 여부는 그룹 기본 설정과 과제별 설정에서 정합니다.
21. deadline 이후 제출이 허용된 상태에서 제출하면 `지각`으로 표시합니다.
22. 과제 삭제 시 제출 코드, 댓글, 리뷰, AI 분석 결과, 집단 코드 비교 분석 결과를 모두 함께 삭제합니다.
23. 과제 삭제 확인 문구는 `삭제하시겠습니까?`로 통일합니다.
24. 삭제된 과제에 대한 기존 알림을 클릭하면 `관련 페이지가 삭제되었습니다` 메시지를 보여줍니다.
25. 과제 삭제자는 별도로 기록하지 않습니다.

### 5.3 코드 제출

1. 그룹원이 과제에 접근합니다.
2. 그룹원이 풀이 코드를 텍스트로 붙여 넣습니다.
3. 코드 입력 필드는 200KB를 넘지 않도록 입력 단계에서 제한합니다.
4. 파일 업로드는 지원하지 않습니다.
5. 클라이언트는 키워드 기반 언어 감지로 우선 syntax highlighting을 적용합니다.
6. AI 분석이 허용된 경우 AI가 언어를 재검증합니다.
7. 사용자는 자동 판단된 언어를 직접 변경할 수 있습니다.
8. 사용자가 언어를 변경하면 현재 제출을 수정하는 것이 아니라 별도 제출로 분리됩니다.
9. 같은 언어로 다시 제출해도 별도 제출로 만들 수 있습니다.
10. 하나의 제출 안에서 코드 내용을 수정할 때마다 해당 제출의 수정 내역을 버전관리합니다.
11. 제출이 완료되면 캘린더의 해당 날짜 제출 상태가 갱신됩니다.
12. 한 사용자는 같은 과제에 여러 번 제출할 수 있습니다.
13. 제출은 사용자별 제출 회차와 제출 시각을 가집니다.
14. 제출 이름 기본값은 `작성자명 + 의 풀이 #N` 형식입니다.
15. 사용자는 제출 이름을 직접 붙이거나 수정할 수 있습니다.
16. 예시는 `A의 1번째 제출`, `B의 1번째 제출`, `A의 2번째 제출`, `C의 1번째 제출`처럼 기록됩니다.
17. 제출에는 코드 설명용 메모(`Markdown`)를 함께 저장할 수 있습니다.
18. 메모는 제출 단위 단일 본문이며 버전 관리를 하지 않습니다. 저장 시 기존 메모를 덮어씁니다.
19. 제출 상세 화면에서는 메모를 버전 목록과 댓글 사이에 표시하고, 작성자만 연필 버튼으로 수정할 수 있습니다.
20. 과제 설정에 따라 AI 분석 작업이 생성됩니다.

### 5.4 AI 분석

AI는 다음 큰 흐름을 담당합니다.

- 문제 URL 분석. 과제에 등록된 문제 URL의 응답 HTML을 받아 제목, 플랫폼, 난이도, 문제 알고리즘 태그 같은 메타데이터를 추론합니다.
- AI 코드 리뷰. 제출된 코드를 검토하고 그룹의 다른 사용자처럼 댓글과 코드 리뷰를 작성합니다.
- 과제 집단 코드 비교 분석. 마감이 지난 과제에 대해 각 제출의 **스냅샷된 원문 코드**(`submission_version`)를 기준으로, **단일 LLM JSON**으로 제출별 역할 구역(`regions`)·Markdown 리포트만 생성합니다. 코드 본문은 LLM 출력에 포함하지 않고 DB 원문과 조인해 응답합니다. 상세 UI는 전용 페이지(`/groups/.../assignments/.../cohort`)에서 칩·가로 코드 열로 표시합니다(5.4.5).

AI 결과는 사람이 검증해야 하는 보조 정보로 취급합니다. 정답 판정, 표절 판정, 실력 평가를 AI가 단독으로 확정하지 않습니다.

#### 5.4.1 문제 URL 분석

1. 사용자가 과제에 문제 URL을 등록합니다.
2. 백엔드 또는 worker가 해당 URL에 직접 HTTP 요청을 보내 응답 HTML을 받아옵니다. 클라이언트 브라우저에서 직접 가져오지 않습니다. 4xx/5xx, 타임아웃, 봇 차단 페이지 등으로 정상 HTML을 얻지 못하면 자동 재시도 없이 즉시 `PROBLEM_ANALYSES.status='FAILED'`로 기록하고 사용자에게 "수동으로 메타데이터를 입력해 주세요" 알림을 보냅니다.
3. 정상 HTML을 받은 경우 받아온 HTML과 URL 자체를 LLM 입력으로 사용해 제목, 플랫폼(`BOJ`, `Programmers`, `LeetCode`, `기타`), 난이도, 문제 알고리즘 태그, 풀이 힌트를 추론합니다.
4. 문제 본문은 저작권 문제 때문에 저장하거나 화면에 그대로 표시하지 않습니다. 저장 대상은 링크, 플랫폼, 제목, 난이도, 추론된 문제 알고리즘 태그, 분석 상태 같은 메타데이터로 제한합니다.
5. 화면에는 항상 "문제 확인하러 가기" 외부 링크 버튼을 두어 사용자가 원본 문제를 직접 열도록 유도합니다.
6. 사용자는 추론된 메타데이터 값을 수동으로 수정할 수 있습니다.
7. 문제 URL 분석이 실패하면 사용자가 제목, 난이도, 플랫폼, 알고리즘 태그를 직접 입력할 수 있습니다. 일부 사이트는 봇 차단 등으로 응답이 비정상일 수 있으므로 수동 입력 fallback은 항상 제공합니다.
8. 직접 입력 메타데이터와 AI 추론 메타데이터를 별도 source로 나누지 않습니다.
9. AI가 지원 플랫폼을 판단하지 못한 외부 URL은 `기타` 플랫폼으로 저장합니다.
10. 문제 분석 재실행은 그룹장과 그룹 관리자만 가능합니다.

#### 5.4.2 AI 코드 리뷰

AI 코드 리뷰는 그룹 규칙 `useAiFeedback`이 `true`인 그룹에서만 동작합니다. AI는 GitHub 봇처럼 동작하며, 결과를 별도 패널이 아닌 일반 댓글·코드 리뷰 시스템에 직접 작성합니다.

1. `useAiFeedback`이 `true`인 그룹에서는 제출 작성자, 그룹장, 그룹 관리자가 명시적으로 "AI 리뷰 요청" 버튼을 눌렀을 때만 AI 코드 리뷰 작업이 큐에 발행됩니다. 새 제출 또는 새 코드 버전이 자동으로 AI 리뷰를 트리거하지 않습니다.
2. worker는 LLM에 제출 코드를 입력하고 다음을 받아옵니다.
   - 코드의 특정 줄 또는 범위에 대한 인라인 리뷰.
   - 코드 일부를 어떻게 고치면 좋은지에 대한 diff 형태(`+`, `-`)의 개선 제안.
   - 제출 전체에 대한 요약 코멘트(인라인 리뷰가 있을 때는 짧은 안내 한 줄로 끝내지 않고, 풀이 전반에 대한 정성적 총평을 자유 서술하도록 프롬프트에서 유도한다).
   - 과제의 문제 URL을 즉시 fetch해 문제 본문 핵심(`요약`, `입력`, `출력`)을 추출하고, 코드·메모와 함께 프롬프트에 포함합니다. 추출한 문제 본문 원문은 DB에 저장하지 않고 요청 처리 후 즉시 폐기합니다.
   - 응답 톤은 제안형·겸손(명령형 금지, 단정적 질책 완화, 본 분석이 틀릴 수 있음을 염두)하도록 시스템 프롬프트에서 제한한다.
   - 인라인·요약 본문은 마크다운 제목(`#`~`####`)·리스트·코드 펜스 등 GFM을 적극 활용하고, 긴 문단 안에서는 `<br />`로 호흡을 주되 ATX 제목·펜스 블록 경계는 빈 줄로만 구분하도록 프롬프트에서 구분한다.
   - 시간·공간 복잡도·연산량·점근식·문제 수식 등 **정량 서술**이 나오면 LaTeX 마크다운(인라인 달러 한 쌍, 디스플레이 이중 달러 블록, 필요 시 펜스 언어 `math`)으로만 쓰도록 프롬프트에서 **강제에 가깝게** 요구한다(평문·`1e9` 나열식 금지). 클라이언트는 해당 본문을 KaTeX로 렌더링한다.
   - 긴 코드·루프·블록은 인라인 백틱이 아니라 fenced 블록만 쓰고, 앞에는 `<br />`·빈 줄·볼드 도입 한 줄·빈 줄·펜스 시작 순으로 두도록 프롬프트에서 고정한다. 뜻 강조는 볼드로만 하고 작은따옴표 의사 강조는 피한다.
   - 심각한 오류나 명백한 시간 제한 위험이 입증되지 않았거나 개선 여지가 크지 않으면 시간 초과·알고리즘·복잡도 관련 코멘트를 내지 않도록 유도한다. 제출이 전반적으로 무난하면 긍정적 총평도 허용한다.
   - 인라인 리뷰는 LLM이 준 앵커 문자열이 제출 코드 줄과 매칭될 때만 코드 옆에 표시된다. 매칭에 실패하면 해당 본문은 AI 튜터의 제출 단위 댓글에 이어 붙이고, 요약 문구는 `lineComments`가 비어 있을 때 인라인 존재를 암시하지 않도록 프롬프트에서 제한한다.
3. worker는 받아온 결과를 우리 댓글·코드 리뷰 시스템에 사용자처럼 등록합니다. AI 작성자임을 식별하기 위해 작성자 표시는 단일 시스템 사용자 `AI 튜터`를 사용합니다(닉네임 `AI 튜터`, 시스템 봇 플래그 `is_system_bot=true`, 프로필 이미지는 정적 SVG 1개를 `fe/public/icons/ai-tutor.svg`에 두고 모든 봇 댓글에서 그대로 사용. AI 로봇 느낌의 이모지 스타일 아이콘으로 통일).
4. 등록되는 형태.
   - 인라인 리뷰는 일반 코드 리뷰와 동일한 데이터 구조로 저장하고, 줄 위치, 범위, 본문을 가집니다.
   - 개선 제안은 본문 안에 코드 블록과 diff 표기를 포함한 일반 댓글 또는 코드 리뷰로 저장합니다.
   - 전체 요약은 제출 단위 댓글로 저장합니다.
5. AI 결과는 사람이 단 댓글·리뷰와 같은 자리에 시간순으로 섞여 보입니다. 작성자 라벨에 `AI 튜터` 배지를 함께 노출합니다.
6. AI 댓글·리뷰 삭제는 그룹장 또는 그룹 관리자만 가능합니다. 일반 그룹원은 답글을 달 수는 있지만 본문 삭제는 할 수 없습니다.
7. AI 코드 리뷰는 제출의 특정 버전(`submission_version_id`)에 귀속됩니다. 새 코드 버전이 만들어지면 이전 버전의 AI 댓글은 자동 삭제되거나 새 버전에 매핑되지 않습니다. 그대로 보존되며, 작성 시각으로 구분됩니다.
8. 같은 버전에 대해 AI 피드백은 **한 번만** 요청할 수 있습니다. 이미 해당 버전에 AI 튜터 댓글 또는 인라인 리뷰가 있으면 버튼이 비활성화되고 API도 거부합니다. 그룹장/관리자가 해당 봇 콘텐츠를 삭제해 더 이상 남아 있지 않으면 같은 버전에 대해 다시 요청할 수 있습니다.
9. AI 토큰은 LLM 호출이 성공해 댓글이 등록된 경우에만 차감합니다. 차감량은 선택한 provider 응답 `usage.total_tokens`(입력+출력 토큰 합)을 그대로 사용합니다. 실패·타임아웃·중간 취소는 0 차감입니다.
10. AI 호출 실패 시에는 worker가 자동으로 1회만 재시도합니다(짧은 backoff). 재시도까지 실패하면 시스템 알림으로 통지하고 사용자가 명시적으로 재트리거할 수 있습니다.
11. AI 코드 리뷰 완료/실패 알림은 제출 작성자 1명에게만 보냅니다. 트리거를 누른 사람이 작성자가 아니라 그룹장/그룹 관리자였더라도 별도 알림을 보내지 않습니다.
11. `useAiFeedback`이 `false`인 그룹에서는 AI 튜터가 어떤 댓글도 작성하지 않으며 "AI 리뷰 요청" 버튼도 노출하지 않습니다. 사용자는 평소처럼 키워드 기반 언어 감지를 통해 syntax highlighting을 받고, 알고리즘 태그 등 보조 메타데이터는 직접 입력합니다(자세한 내용 5.4.3).
12. 토큰 부족 또는 호출 한도 같은 운영 상태로 AI 작성이 실패하면 사용자에게 명확한 메시지를 보여주고 재시도를 허용합니다.
13. AI 피드백 프롬프트는 계속 테스트하면서 수정할 예정이므로 기능별로 쉽게 관리할 수 있어야 합니다. 프롬프트 관리 단위는 문제 분석, 코드 리뷰, 집단 코드 비교처럼 기능별로 나눕니다.
14. AI 튜터에게 보낸 답글에 대해 AI 튜터는 다시 응답하지 않습니다. 답글에 의한 알림은 일반 사용자 답글과 동일하게 다른 참여자에게만 보냅니다.
15. AI 코드 리뷰에서 문제 URL 요청을 보낼 때는 고정 UA를 쓰지 않고 요청마다 User-Agent를 랜덤 선택해 보냅니다.

#### 5.4.3 AI 피드백 비활성 시 사용자 입력

`useAiFeedback`이 `false`인 그룹에서는 일부 메타데이터를 사용자가 직접 채웁니다.

- 제출 언어. 키워드 기반 언어 감지 결과를 기본값으로 두고 사용자가 변경할 수 있습니다.
- 제출 알고리즘 태그. 과제(문제) 메타데이터의 알고리즘 태그를 기본값으로 미리 채워 줍니다. 사용자는 본인의 풀이 특성에 맞게 수정할 수 있습니다.
- 개선 포인트 텍스트는 AI 전용 정보이므로 비활성 그룹에서는 표시하지 않습니다.

#### 5.4.4 제출 코드 기계 번역(미구현)

그룹 공통 번역 규칙, `SUBMISSION_TRANSLATIONS` 캐시, 제출 화면의 "번역 보기" API는 **현재 제품 범위에 포함하지 않습니다**. 제출 언어는 제출 폼에서 선택하는 `language`만 사용합니다.

#### 5.4.5 과제 집단 코드 비교 분석

그룹원이 서로 제출한 **원문 코드 전체**를 동일한 표면에서 비교하기 위한 기능입니다. 집단 분석 파이프라인은 과제 `problem_url`을 서버가 fetch해 **제출 AI 코드 리뷰와 동일한 규칙**으로 정제한 요약·입력·출력을 **LLM 입력으로만** 넘기며, 문제 원문 전체를 서비스 화면에 그대로 노출하지 않습니다. 리포트는 문제 요구 대비 **제출 간** 유사점·차이를 서술합니다. 정제 결과는 DB에 별도 영구 저장하지 않고 처리 중 메모리로만 씁니다. 결과는 학습용 보조 정보이며, 정답·표절·실력 평가의 단독 근거로 쓰이지 않습니다.

**전제·활성 조건**

1. 과제 **마감이 지난 뒤**에만 분석을 실행할 수 있습니다.
2. 분석 집합에 포함할 제출은 **지각 제출**(`allow_late_submission` 등 과제·그룹 규칙에 따른 지각)도 포함합니다.
3. 제출이 **2개 미만**이면 분석을 실행할 수 없습니다(0개·1개는 불가).

**처리 파이프라인(LLM 단일 경로)**

1. 파이프라인 입력에는 과제 `problem_url`에서 추출한 문제 본문(`problemContext`, 실패 시 null)·URL 문자열과 각 제출의 **최신 유효 버전** 원문을 **`source`(스냅샷 그대로)** 및 줄 단위 **`lines[]`**(`split('\n')`, 빈 줄은 빈 문자열 요소)·메타(제출 ID, 언어 등)가 포함됩니다(`lines.join('\n') === source`로 줄 번호 계약을 고정). 저장된 `artifacts` JSON에는 코드 본문을 넣지 않고, 완료 후 조회 시 `ASSIGNMENT_COHORT_ANALYSIS_MEMBERS.submission_version_id`로 `SUBMISSION_VERSIONS`를 조인해 `code`·`language`를 채웁니다.
2. LLM은 **역할 구역 라벨링(`regions`)·Markdown 리포트(`reportMarkdown`)**만 구조화된 JSON으로 반환합니다. `regions`는 줄번호 대신 `startAnchorText`·`endAnchorText`를 내고, 서버가 원문 `source`에서 앵커를 찾아 줄범위를 계산합니다. 제출마다 구역 개수·`roleId` 집합은 **코드 구조에 맞게 독립적**일 수 있으며, 서버는 제출 간 라벨을 통일하지 않습니다. `reportMarkdown`은 `#`~`###` 목차, 목록·굵게 표시·코드 펜스 발췌 등 **보고서 형태**를 갖추도록 프롬프트에서 요구합니다(HTML `<table>`·Markdown 파이프 표 등 **표 형식은 쓰지 않음** — 렌더 불안정 방지). 프론트는 마크다운 렌더 시 **문단 간 가독성**을 위해 제한적으로 `<br />`(rehype-sanitize 적용)를 허용합니다. 파싱·검증(앵커 매칭, 구역 줄 범위 클램프, 제출당 구역 수·중복·최소 줄 수 등)에 실패하면 분석 전체가 실패(`FAILED`)입니다. 리포트 본문은 제출을 언급할 때 `[[SUBMISSION:<uuid>]]` 플레이스홀더만 쓰고, 화면에서는 제목·버전·프로필 칩으로 치환합니다.
3. 제출마다 프로그래밍 언어가 달라도 됩니다. 리포트는 **여러 언어를 혼합한 코드 펜스**로 발췌를 서술할 수 있으며, 단일 목표 언어로 통일할 것을 요구하지 않습니다.
4. 핵심 로직 왜곡 없이 비교 서술하는 것을 목표로 하되, 구체적 스타일 통일 규칙은 프롬프트로 안내합니다.
5. 루프·시뮬레이션 역할 구역은 프롬프트에서 **`for`/`while`/`do` 헤더가 포함된 연속 줄 구간**으로 잡도록 강제하여, 닫는 꼬리·함수 앞부분만 칠하는 오류를 줄인다.

**데드 코드 표시**

1. 선언만 있고 사용되지 않는 변수·함수 등은 **정적 분석**으로 판별합니다.
2. 해당 구간은 UI에서 강조(예: 빨간 표시)할 수 있으나, **집단 비교의 역할 구역 판단에는 반영하지 않습니다**.

**산출물**

1. **제출별 DB 스냅샷 원문 코드**를 역할(`roleId`)별 배경색으로 구역 표시합니다. 색은 **해당 제출 패널 안에서만** `roleId` 문자열 목록을 정렬해 매깁니다. 패널 간에 같은 문자열 `roleId`가 있으면 동일 색으로 연동 강조(호버)에 쓰이지만, 제출마다 `roleId` 집합이 다를 수 있습니다. 가로 스크롤 다열과 양쪽 더미 패딩으로 첫·마지막 열을 뷰포트 중앙에 맞출 수 있습니다.
2. 과제 전체를 아우르는 **Markdown 리포트 1개**와 제출 참조용 플레이스홀더 규약을 둡니다.
3. 리포트 **내보내기**(PDF 등)는 1차 범위에 포함하지 않습니다.
4. 이 기능과 관련하여 **모델명·프롬프트 버전·시드 등 LLM 구성 메타는 어떠한 저장소에도 기록하지 않습니다**(사용자 리포트·DB·운영 로그 포함). 미묘한 판단 근거는 Markdown 본문으로만 서술합니다.

**실행·락·재시도**

1. **과제당 성공한 분석은 1회만** 허용합니다. 한 과제에 대해 최종 상태가 `DONE`이 되면 같은 과제에서 다시 분석을 실행할 수 없습니다.
2. **성공(`DONE`)으로 확정되기 전**까지는 “아직 분석 전”으로 봅니다. 이 구간에서 **새로 제출되거나 갱신된 코드**는 **다음 실행 시도의 입력 집합에 포함**됩니다(실패 후 재시도 시에도 최신 유효 제출 기준으로 다시 수집).
3. **성공 확정 이후**에 만들어진 제출·새 코드 버전은 **이번 과제 분석 결과에 포함되지 않습니다**.
4. 작업이 실패(`FAILED`)한 경우에는 **재시도 가능**합니다. 토큰 차감 등 비용 규칙은 구현 시 기존 AI 작업과 동일한 원칙(성공 시 차감 등)을 따릅니다.

**공개 범위**

1. 그룹원 상호간 **원문 코드·작성자 실명**이 100% 공개되는 전제입니다. 가입·그룹 문화·약관에서 이를 전제로 안내합니다.

**지표**

1. 기능 성공 여부는 **정성**(만족도·인터뷰)과 **정량**(예: 완료율·토큰·처리 시간 등)을 함께 봅니다.

### 5.5 댓글과 코드 리뷰

1. 과제 단위 댓글과 제출물 단위 댓글을 모두 지원합니다.
2. 과제 단위 댓글은 과제 전체에 대한 일반 댓글 느낌으로 사용합니다.
3. 제출물 단위 댓글은 코드 리뷰 느낌으로 사용합니다.
4. 같은 그룹원은 모두 코드 리뷰를 남길 수 있습니다.
5. 그룹원은 제출 코드의 특정 부분을 드래그해서 코드 리뷰를 남길 수 있습니다.
6. 코드 리뷰 UX는 GitHub 코드 리뷰처럼 특정 줄 또는 범위, 파일 전체, 제출 전체에 코멘트를 붙이는 형태를 지향합니다. 줄 범위 리뷰의 스레드 UI는 참조 범위의 **마지막 줄** 아래에 표시합니다.
7. 코드 리뷰가 달린 제출을 작성자가 수정하면 같은 제출 안에 새 버전이 생성됩니다.
8. 제출 버전 간 차이는 diff로 계산하고, 추가와 삭제를 `+++`, `---` 같은 방향으로 표현합니다. (```diff 처럼)
9. 리뷰를 받은 사용자의 해결됨 표시는 제공하지 않습니다.
10. 댓글이나 리뷰가 생성되면 대상 사용자에게 알림이 갑니다.
11. `@` 멘션을 지원하고, 멘션된 사용자에게도 알림을 보냅니다.
12. 작성자는 본인의 댓글과 리뷰를 수정하거나 삭제할 수 있습니다.
13. 작성자가 본인의 댓글이나 리뷰를 삭제하면 실제 삭제합니다.
14. 그룹장 또는 그룹 관리자가 다른 사람의 댓글이나 리뷰를 삭제하면 본문을 `삭제된 댓글입니다`로 대체합니다.
15. 관리자 삭제 시 별도 삭제 사유는 입력하지 않습니다.
16. 관리자 삭제로 `삭제된 댓글입니다`가 된 댓글의 답글은 그대로 유지합니다.
17. 관리자 삭제된 댓글은 작성자 이름과 작성 시각을 남깁니다.
18. 리뷰에 답글이 달리면 제출 작성자, 기존 리뷰 작성자, 멘션된 사용자에게 알림을 보냅니다.
19. `@` 멘션 대상은 같은 그룹의 그룹원으로 제한합니다.
20. 댓글과 리뷰는 수정 이력을 보여주지 않고 `(수정됨)`만 표시합니다.
21. 제출물 전체 댓글과 드래그 기반 코드 리뷰는 같은 스레드 UI에 보여줍니다.
22. 제출 코드에 대한 일반 댓글과 코드 라인 리뷰는 같은 입력 UI로 처리합니다.
23. 리뷰 작성 후 수정 가능 시간 제한은 두지 않습니다.
24. 스레드 정렬 기본값은 오래된 순입니다.
25. 제출 삭제 시 해당 제출에 달린 댓글과 리뷰도 함께 삭제된다는 안내를 보여줍니다.

### 5.5.1 답글 (대댓글)

1. 모든 댓글 도메인(과제 댓글, 제출 댓글, 코드 리뷰, 공지 댓글, 커뮤니티 댓글)은 같은 형태의 답글 모델을 가집니다.
2. 답글 깊이는 1단으로 제한합니다. 답글에 다는 답글도 같은 부모 스레드 아래에 평면적으로 쌓입니다(트위터 스타일).
3. 코드 리뷰 답글은 별도 테이블 `REVIEW_REPLIES`에 저장하고, 일반 댓글 답글은 같은 테이블의 `parent_comment_id`를 사용합니다.
4. 답글 작성자는 본인 답글을 수정·삭제할 수 있고, 그룹장/그룹 관리자는 다른 사람의 답글을 `삭제된 댓글입니다`로 가립니다.
5. 답글 알림 정책은 5.5절의 댓글 알림 정책과 동일합니다(원 댓글 작성자 + 멘션 + 제출 작성자).
6. 답글 UI는 부모 카드 아래에 들여쓰기된 스레드 형태로 보여주고, 부모 카드의 "답글 달기" 버튼이 인라인 입력창을 토글합니다.

### 5.5.2 이모지 반응

1. 모든 댓글과 답글에 이모지 반응을 달 수 있습니다(과제 댓글, 제출 댓글, 코드 리뷰, 코드 리뷰 답글, 공지 댓글, 커뮤니티 댓글).
2. 이모지는 고정 세트가 아니라 자유 이모지 선택입니다. 사용자는 picker에서 임의의 유니코드 이모지를 고를 수 있습니다.
3. 같은 사용자가 같은 댓글에 같은 이모지를 두 번 달 수 없습니다(`(target, user, emoji)` unique). 이미 단 이모지를 다시 누르면 토글로 해제됩니다.
4. 같은 사용자가 같은 댓글에 서로 다른 이모지는 여러 개 달 수 있습니다.
5. UI는 GitHub 스타일로 카드 하단에 이모지별 카운트와 누가 달았는지 hover 툴팁을 보여줍니다.
6. 이모지 반응은 알림 이벤트를 발행하지 않습니다(노이즈 방지).
7. 이모지 반응을 단 사용자가 탈퇴하더라도 카운트는 그대로 유지하고, 본인이 단 행만 사라집니다.
8. AI 봇이 작성한 댓글·리뷰에도 사용자가 이모지를 달 수 있습니다. 봇은 이모지를 달지 않습니다.

### 5.5.3 코드 리뷰 인라인 UI

1. diff 화면의 인라인 코드 리뷰는 GitHub 스타일 카드(아바타·작성자·시간·본문·이모지·답글)로 표시합니다(`fe/src/ui/comments/CommentCard.tsx`).
2. 리뷰가 다중 라인 범위(`startLine`–`endLine`)인 경우, 해당 범위에 속한 모든 new 라인의 행 배경에 primary 색을 14~18% 섞은 옅은 음영을 칠해 한 덩어리로 보이게 합니다. 별도의 세로 막대(border) 시각 강조는 두지 않습니다.
3. 인라인 리뷰 카드는 접기 가능합니다. 접힌 상태에서는 작성자 프로필 아바타 칩만 같은 자리에 남고, 답글이 있으면 옆에 `+N` 카운트가 따라붙습니다. 칩을 누르면 다시 카드 전체가 펼쳐집니다. 접힘/펼침 상태는 클라이언트 한정으로 유지하며 새로고침 시 펼쳐진 기본값으로 복원됩니다.
4. 본문에 fenced code block(```lang … ```) 가 포함된 경우 토큰 단위 syntax highlighting을 적용합니다(언어 미지정 시 highlight 없이 monospace로만 출력).
5. 코드 블록의 시각적 테두리는 한 겹만 둡니다(외곽 테두리 + 내부 `pre` 무테두리).

### 5.6 피드백 공개 정책

피드백 공개 정책은 단순화합니다.

- 그룹원은 같은 그룹의 모든 제출 코드, 사람 댓글, 코드 리뷰, AI 코드 리뷰를 볼 수 있습니다.
- 제출 단위 또는 과제 단위로 별도의 공개 범위 토글은 두지 않습니다.
- 그룹 기본 정책 + 과제별 override 매트릭스는 두지 않습니다.
- 다만 AI 코드 리뷰 자체를 만들지 말지는 그룹 규칙 `useAiFeedback`이 결정합니다(5.1.1).
- deadline 이전·이후로 공개 범위가 달라지는 분기도 두지 않습니다.

스포일러 방지를 원할 경우 그룹장은 `useAiFeedback`을 끄거나 새 과제 생성을 마감일 직전으로 미루는 식으로 운영적으로 대응합니다. 시스템은 단일한 "그룹원 모두 공개" 모델만 유지합니다.

### 5.7 그룹 캘린더

그룹 캘린더는 다음 정보를 보여줍니다.

- 날짜별 과제.
- 각 과제의 deadline.
- 구성원별 제출 여부.
- 본인의 제출 상태.
- 과제 상태.
- 주간 캘린더와 월간 캘린더 전환.

주간과 월간 캘린더에서는 같은 날짜의 과제를 모두 표시합니다. 데스크톱에서는 셀에 제목 말줄임으로 표시하고 모바일에서는 점으로 요약합니다. 날짜 셀을 누르면 해당 날짜 마감 과제 목록 모달이 열리며, 모달에서 과제 상세로 이동합니다. 달력 UX는 범용 캘린더 제품들의 패턴을 충분히 참고합니다.

캘린더 헤더에는 필터 버튼을 제공하며, 제출자(복수 선택), 해결/미해결, 출처(플랫폼), 알고리즘 태그, 제목 검색 조건을 조합해 캘린더 셀과 날짜 모달의 과제 목록을 함께 필터링합니다.

과제 상태는 다음과 같습니다.

- 예정.
- 진행 중.
- 제출 마감.
- 종료.

### 5.8 공지와 커뮤니티

그룹에는 공지와 커뮤니티 영역이 존재합니다.

- 공지는 그룹장 또는 그룹 관리자가 작성합니다.
- 공지는 상단 고정, 중요 표시, 읽음 확인을 지원할 수 있습니다.
- 커뮤니티 게시글은 일반 질문, 풀이 토론, 스터디 운영 대화에 사용합니다.
- 공지와 커뮤니티 댓글도 알림 대상이 됩니다.
- 공지 댓글을 허용합니다.
- 커뮤니티 글은 그룹원도 작성할 수 있습니다.
- 공지와 커뮤니티 글 수정 이력은 보여주지 않고 `(수정됨)`만 표시합니다.
- 공지와 커뮤니티는 같은 게시판 모델로 설계하고 가능한 한 같은 코드를 재사용합니다.
- 공지와 커뮤니티의 차이는 게시글 종류, 고정 여부, 중요 표시 여부 같은 최소 필드로 표현합니다.
- 공지와 커뮤니티는 파일 첨부와 이미지 첨부를 MVP에서 지원하지 않습니다.
- 공지와 커뮤니티 본문은 Markdown을 지원합니다.
- Markdown code block도 지원합니다.
- 공지 댓글, 커뮤니티 댓글, 리뷰 답글 모두 `@` 멘션을 지원하고 멘션된 사용자에게 알림을 보냅니다.

### 5.9 후속 릴리스 후보 AI 기능

다음 기능은 1차 릴리스 범위 밖이며 후속 릴리스에서 다시 검토합니다.

- 코드 주석 자동 생성.
- 수도 코드 변환.

제출 코드 기계 번역(5.4.4)은 현재 구현 범위에 포함하지 않습니다.

AI 분석 결과가 틀렸거나 충분하지 않으면 사용자가 재실행할 수 있습니다. 재실행 결과는 기존 결과를 덮어쓰며, 사람이 검증해야 하는 보조 정보로 취급합니다.

### 5.10 그룹 대시보드

그룹 내부 대시보드는 그룹원의 문제 풀이 활동을 한 화면에서 확인하는 통계 화면입니다.

1. 그룹 탭에 `대시보드`를 추가합니다.
2. 대시보드는 기간 필터(7일, 30일, 90일)를 제공합니다.
3. 상단 KPI 카드에 다음 값을 표시합니다.
   - 총 제출 수.
   - 활동 멤버 수.
   - 그룹 참여율.
   - 지각률.
4. 그래프는 최소 2종을 반드시 포함합니다.
   - 일자별 제출 추이를 보여주는 꺾은선 그래프.
   - 멤버 기여 비율을 보여주는 원형 그래프.
5. 멤버별 지표 테이블을 제공합니다.
   - 해결 과제 수.
   - 제출 수.
   - 온타임률.
   - 활동일 수.
   - 연속 제출일 수.
6. 과제 건강도 섹션을 제공합니다.
   - 과제별 참여율.
   - 과제별 지각률.
   - 미제출 인원 수.
7. 플랫폼 분포 섹션을 제공합니다.
8. 지표 계산은 기존 그룹/과제/제출 데이터를 조합해 계산합니다.
9. 대시보드는 관리자 전용이 아니라 그룹원 전체가 조회할 수 있습니다.

## 6. 과제 설정 초안

그룹장 또는 그룹 관리자는 과제에서 다음 값을 설정할 수 있습니다.

- 문제 링크.
- 과제 제목.
- 과제 설명 또는 메모.
- 문제 출처.
- 문제 난이도.
- 문제 분석 상태.
- 문제 알고리즘 태그.
- 문제 플랫폼 태그.
- 문제 메타데이터.
- 과제 날짜.
- 제출 deadline.
- deadline 이후 제출 허용 여부(과제별 override).

과제별 피드백 공개 override, AI 세부 토글 override는 두지 않습니다. AI 코드 리뷰 활성 여부는 그룹 규칙 `useAiFeedback`으로만 관리합니다(5.1.1).

그룹 기본 설정은 다음 값을 가집니다(5.1.1과 동일 항목, 단일 source).

- `useDeadline`: 마감일 사용 여부.
- `defaultDeadlineTime`: 기본 마감 시각(KST `HH:MM`).
- `allowLateSubmission`: 마감 이후 제출 허용 여부.
- `useAiFeedback`: AI 코드 리뷰 작성 활성 여부.
- `allowEditAfterSubmit`: 제출 후 코드 수정 허용 여부.
- `assignmentCreatorRoles`: 과제 생성 가능 역할 범위(`OWNER_ONLY` | `OWNER_AND_MANAGER`).

그룹 관리자는 다음 권한을 가집니다.

- 그룹원 초대.
- 그룹원 강퇴.
- 과제 생성(그룹 규칙 `assignmentCreatorRoles`가 `OWNER_AND_MANAGER`일 때).
- 과제 수정.
- 과제 삭제.
- 댓글 삭제.
- 리뷰 삭제.
- AI 봇이 작성한 댓글·리뷰 삭제.
- 그룹원을 그룹 관리자로 승격.
- 그룹 관리자를 그룹원으로 변경.
- 가입 신청 승인·거절.
- 이메일 초대 발송.

그룹장 전용 권한은 다음과 같습니다.

- 그룹 삭제.
- 그룹장 위임.
- 그룹 코드 갱신.

- 가입 신청 처리 결과(신청자 대상).
- 이메일 초대 수락 결과(발송자 대상).

### 7.1 수신자 정리(구현 기준)

| 이벤트 | 수신자 | 비고 |
|--------|--------|------|
| 새 과제 등록(`ASSIGNMENT_CREATED`) | 해당 그룹의 **활성 멤버 전원**, 단 **과제를 등록한 사람은 제외** | 인앱 문구: `{그룹명} 그룹에서 "{과제명}" 과제가 등록되었습니다.` 행위자 아바타 없음. 홈·`GET /api/v1/users/me/notifications`에 노출 |
| 제출물에 **일반 댓글**(루트 `Comment`) | **제출 작성자**(`COMMENT_ON_MY_SUBMISSION`), 본인 제외 | 제출 상세 스레드 댓글 |
| 제출물 댓글에 **답글** | **부모 댓글 작성자**(`REPLY_ON_MY_COMMENT`), 본인 제외 | 1단 답글만 |
| diff **인라인 코드 리뷰**(`Review` 생성) | **제출 작성자**(`REVIEW_ON_MY_SUBMISSION`), 본인 제외 | 인앱 문구는 사용자에게 **댓글**로 통일(코드 리뷰라는 용어는 알림 제목에 쓰지 않음). 구 저장 데이터는 API 응답 시 제목을 치환할 수 있음. |
| 코드 리뷰에 **답글**(`ReviewReply`) | (1) **리뷰 작성자**(`REPLY_ON_MY_REVIEW`) — 단 **AI 튜터(시스템 봇) 리뷰는 리뷰 작성자 쪽 알림 생략** (2) 리뷰 작성자가 아닌 **제출 작성자**(`REPLY_ON_REVIEW_I_WROTE`) — 본인 제외, 리뷰어=제출자면 중복 없음 | `design.md` 5.5.1과 동일 정책 |

멘션·이모지 반응 등은 위 표에 없는 기존 설계 문단을 따릅니다.

알림은 인앱 알림으로 시작합니다.

- 상단 또는 전역 내비게이션에 알림 진입점을 둡니다.
- 읽지 않은 알림이 있으면 빨간색 점 안에 숫자로 unread count를 표시합니다.
- 알림은 읽음과 안읽음 상태를 가집니다.
- 알림 페이지(`/notifications`)에 알림 목록을 표시합니다. 홈의 최근 알림 카드에서 `모두 보기`로 같은 페이지로 들어올 수 있습니다.
- 알림 문구 예시는 `누구누구님이 댓글을 달았습니다`처럼 행위자와 행동이 드러나는 형태로 작성합니다.

- 알림은 사용자가 삭제하기 전까지 보관합니다.
- 알림 TTL은 두지 않습니다.

- 알림 삭제는 개별 삭제와 전체 삭제를 모두 지원합니다.
- 외부 메신저(카카오톡, 디스코드, 슬랙) 연동은 1차 릴리스 범위 밖입니다.

## 9. 확정된 세부 정책

### 9.1 그룹 설정

1. 그룹명 길이 제한은 20글자로 확정합니다.
2. 그룹 설명은 plain text 최대 500자입니다.
3. 그룹 최대 인원수 기본값은 10명, 최소값은 2명, 시스템 한도는 50명입니다. 운영 중에는 현재 멤버 수보다 작게 줄일 수 없습니다.
4. 그룹 코드는 8자, 영문 대소문자 + 숫자, 대소문자 구분으로 그룹당 1개입니다. 그룹장만 갱신할 수 있고, 갱신 시 이전 코드와 이전 코드로 만들어진 초대 링크는 즉시 무효화됩니다. 코드 자체는 그룹원 모두에게 노출합니다.
5. 그룹은 공개/비공개 구분이 없습니다. 공개 검색 UI나 비밀번호 보호 같은 개념도 두지 않습니다.
6. 그룹 가입 방식은 `코드`, `링크`, `가입 신청`, `이메일 초대` 4가지 토글로 관리합니다. 모든 토글이 꺼지면 신규 가입은 동결됩니다. 운영자가 수동으로 멤버를 추가하는 우회 경로는 두지 않습니다.
7. 그룹원은 직접 그룹 탈퇴를 할 수 있습니다. 단, 그룹장은 탈퇴가 아니라 그룹 삭제만 할 수 있습니다.
8. 가입 신청은 정원이 가득한 상태에서도 접수합니다. 승인 시점에 정원이 가득 차 있으면 승인을 거부합니다.
9. 이메일 초대는 그룹당 1시간 내 50건으로 제한합니다. 초과 시 `RATE_LIMITED`로 거부합니다. 같은 주소에 활성 PENDING 토큰이 있으면 기존 토큰을 만료 처리한 뒤 새 토큰을 발급합니다.
10. 그룹 규칙(6항목) 변경 시 그룹원에게 알림을 보내지 않습니다. 변경은 그룹 설정 화면 자체로 확인합니다.
11. AI 코드 리뷰 작성자는 단일 시스템 사용자 `AI 튜터`(`is_system_bot=true`)입니다. AI 댓글·리뷰는 그룹장 또는 그룹 관리자만 삭제할 수 있습니다.

### 9.2 과제 생성

1. 과제 설명 또는 메모는 Markdown 없이 일반 텍스트로만 입력합니다.
2. 과제 삭제 모달 오픈 시점의 스냅샷 기준으로 삭제 영향을 카운트로 보여줍니다. 예시 `제출 12개, 댓글 53개, 리뷰 21개가 함께 삭제됩니다.`
3. 과제 삭제 실행 직전에는 삭제 대상을 실시간 재집계해서 최종 삭제합니다.

### 9.3 제출과 리뷰

1. 코드 라인 리뷰는 제출 버전별로 분리하고, 새 버전에는 이전 버전 라인 리뷰를 보여주지 않습니다.
2. 제출 코드 diff 화면에서 인라인 댓글을 함께 보여줍니다.
3. 제출 코드 diff 화면의 기본 댓글 범위는 현재 선택 버전 댓글만 보여줍니다.
4. diff 인라인 코드 리뷰 스레드는 접어 작성자 아바타 칩만 보이게 할 수 있으며, 클릭하면 다시 펼칩니다(5.5.3).
5. 제출 상세의 본문 댓글도 같은 `CommentCard`로 렌더링하며 답글, 이모지 반응을 동일한 방식으로 지원합니다(5.5.1, 5.5.2).
6. 다중 라인 범위 리뷰는 해당 라인 구간의 행 배경 음영으로만 강조하고, 라인 게이지 바·세로 border는 두지 않습니다.

## 10. 현재 가정

- 문제 본문은 저장하지 않고 문제 링크와 메타데이터만 저장합니다.
- 문제 본문은 저작권 문제 때문에 서비스 화면에 그대로 표시하지 않습니다.
- 문제 링크는 외부 URL만 허용합니다.
- 문제 링크 유효성 검사는 지원 플랫폼별 파싱으로 수행합니다.
- 문제 플랫폼 태그는 `BOJ`, `Programmers`, `LeetCode` 같은 값으로 저장합니다.
- MVP 지원 플랫폼은 `BOJ`, `Programmers`, `LeetCode`로 시작합니다.
- 지원 플랫폼으로 확정되지 않는 외부 URL은 AI가 플랫폼과 문제 정보를 판단합니다.
- AI가 지원 플랫폼을 판단하지 못한 외부 URL은 `기타` 플랫폼으로 저장합니다.
- `기타` 플랫폼 문제도 과제 생성과 코드 제출을 동일하게 허용합니다.
- `기타` 플랫폼 문제는 문제 분석 실패율이 높을 수 있다는 안내 문구를 보여줍니다.
- 같은 문제 링크를 여러 과제에 중복 등록할 수 있으며 중복 경고는 표시하지 않습니다.
- 문제 메타데이터는 문제당 하나의 source만 관리합니다.
- AI 판단으로 메타데이터 초안이 생기고 사용자가 그 값을 수정합니다.
- 코드 실행과 정답 판정은 MVP에서 제외합니다.
- AI는 보조 분석 도구이며 최종 판단자가 아닙니다.
- 그룹 단위의 폐쇄형 스터디를 우선합니다.
- 인앱 알림을 기본 알림 채널로 둡니다.
- 로그인은 Google OAuth와 GitHub OAuth를 지원합니다.
- 로그인은 OAuth만 지원하고 이메일/비밀번호 로그인은 지원하지 않습니다.
- Google OAuth 계정과 GitHub OAuth 계정은 동일 이메일이어도 별도 계정으로 취급합니다.
- 사용자 프로필 이미지는 각 OAuth 제공자에서 받은 이미지를 사용합니다.
- OAuth 계정 탈퇴는 해당 OAuth 계정에만 적용하고, 같은 이메일의 다른 OAuth 계정에는 영향을 주지 않습니다.
- 사용자 프로필 이미지는 로그인 시마다 OAuth 제공자 기준 최신 이미지로 동기화합니다.
- 닉네임 기본값은 OAuth 제공자의 표시명을 사용합니다.
- 사용자는 서비스 내에서 닉네임을 직접 변경할 수 있습니다.
- 닉네임 중복은 허용합니다.
- 그룹 역할은 `그룹장`, `그룹 관리자`, `그룹원` 3단계입니다.
- 그룹을 만든 사용자가 기본 그룹장이 됩니다.
- 그룹은 그룹명, 그룹 설명, 최대 인원수, 그룹 코드, 가입 방식 토글, 그룹 규칙을 가집니다(자세한 항목은 5.1 참조).
- 그룹명 길이 제한은 최대 20글자입니다.
- 그룹 설명은 plain text 최대 500자입니다.
- 그룹 프로필 이미지와 배너 이미지는 1차 릴리스에서 제공하지 않습니다.
- 그룹장 또는 그룹 관리자는 그룹명, 그룹 설명, 최대 인원수, 가입 방식 토글, 그룹 규칙을 수정할 수 있습니다.
- 그룹 코드 갱신은 그룹장만 수행할 수 있습니다.
- 그룹장은 1명만 존재합니다.
- 그룹 관리자는 여러 명일 수 있습니다.
- 그룹장은 그룹 삭제 권한을 가집니다.
- 그룹 삭제 시 그룹명을 입력해야 합니다.
- 그룹 삭제 시 `삭제된 그룹은 복구할 수 없습니다` 경고 문구를 보여줍니다.
- 그룹장이 그룹 삭제를 실행하면 그룹 데이터(과제, 제출, 댓글, 리뷰, AI 분석 결과)는 모두 삭제됩니다.
- 그룹 삭제가 발생해도 기존 알림 데이터는 삭제하지 않습니다.
- 과제 삭제 시 제출 코드, 댓글, 리뷰, AI 분석 결과, **집단 코드 비교 분석 결과**(5.4.5)를 모두 함께 삭제합니다.
- 과제 삭제 확인 문구는 `삭제하시겠습니까?`로 통일합니다.
- 그룹장은 다른 그룹원에게 그룹장 권한을 넘길 수 있습니다.
- 그룹장 권한을 넘길 때 새 그룹장의 수락은 필요하지 않습니다.
- 그룹 삭제와 그룹 코드 갱신을 제외한 그룹 운영 권한은 그룹장과 그룹 관리자가 동일합니다.
- 그룹장과 그룹 관리자는 그룹원을 그룹 관리자로 승격하거나 그룹 관리자를 그룹원으로 변경할 수 있습니다.

- 그룹원은 직접 그룹 탈퇴를 할 수 있습니다.
- 그룹장은 그룹을 탈퇴할 수 없고 그룹 삭제만 할 수 있습니다.
- 탈퇴한 사용자의 제출 코드와 댓글은 이름과 함께 남깁니다.
- 탈퇴한 사용자의 작성자 프로필 링크는 비활성 상태로 보여줍니다.
- 가입 방식은 4가지(`코드 입력`, `초대 링크`, `가입 신청`, `이메일 초대`)를 지원하며 그룹별 4개 토글로 켜고 끕니다.
- 모든 토글이 꺼지면 어떠한 신규 가입도 발생하지 않습니다(가입 동결).
- 그룹 코드는 그룹당 1개의 영구 코드로, 8글자이며 영문 대문자, 영문 소문자, 숫자를 사용합니다.
- 그룹 코드는 대소문자를 구분합니다.
- 그룹 코드와 초대 링크의 만료 시각·최대 사용 횟수 같은 일회성 모델은 두지 않습니다.
- 그룹장이 그룹 코드를 갱신하면 이전 코드와 이전 코드로 만들어진 초대 링크는 즉시 무효화됩니다.
- 무효화된 코드 또는 링크 접근은 `404 NOT_FOUND`로 응답합니다.
- 그룹 코드 입력 실패 횟수 제한이나 추가 잠금 정책은 두지 않습니다.
- 이메일 초대는 1회용 초대 토큰을 메일로 발송합니다. 토큰 TTL은 7일이고 1회 사용 후 만료됩니다.
- 이메일 초대 수신자가 미가입자이면 OAuth 로그인 또는 가입 후 자동으로 그룹에 합류합니다.
- 가입 신청에는 자기소개나 신청 메시지를 받지 않습니다.
- 그룹 관리자는 그룹 삭제와 그룹 코드 갱신을 제외한 그룹 운영 권한을 그룹장과 동일하게 가집니다.
- 그룹 관리자는 그룹원 초대(이메일 발송 포함), 그룹원 강퇴, 과제 수정, 과제 삭제, 댓글 삭제, 리뷰 삭제 권한을 가집니다.
- 그룹 관리자는 그룹 규칙 `assignmentCreatorRoles`가 `OWNER_AND_MANAGER`일 때 과제 생성 권한을 가집니다.
- 그룹원 강퇴는 그룹장과 그룹 관리자만 수행할 수 있습니다.
- 그룹 최대 인원수에 도달한 그룹은 새 멤버 합류를 거부합니다.
- 운영자가 우회로 멤버를 강제로 추가하는 기능은 두지 않습니다.
- 과제 하나는 문제 하나만 가집니다.
- 과제 제목은 문제 분석 결과 제목을 기본값으로 사용하고 그룹장 또는 그룹 관리자가 수정할 수 있습니다.
- 과제에는 간단한 설명 또는 메모 필드를 둡니다.
- 과제 설명 또는 메모는 일반 텍스트만 지원하고 Markdown은 지원하지 않습니다.
- 제출 코드는 텍스트 붙여넣기만 지원하고 파일 업로드는 지원하지 않습니다.
- 한 사용자는 같은 과제에 여러 번 제출할 수 있습니다.
- 제출 이름 기본값은 `작성자명 + 의 풀이 #N` 형식이며 사용자가 직접 이름을 붙이거나 수정할 수 있습니다.
- 제출은 오래된 순을 기본 정렬로 사용합니다.
- 제출은 제출 시각별 정렬, 특정 사용자 필터링, 특정 언어 필터링을 지원합니다.
- 제출 코드 길이 제한은 200KB입니다.
- 제출 코드 입력 필드는 200KB를 넘지 않도록 입력 단계에서 제한합니다.
- 비공개 제출 기능은 제공하지 않습니다.
- 제출 코드는 텍스트와 최종 언어 값을 함께 저장합니다.
- 제출은 수정과 실제 삭제를 지원하며 soft delete는 사용하지 않습니다.
- 제출 후 수정 허용 기본값은 그룹 규칙 `allowEditAfterSubmit`(기본 `true`)을 따릅니다.
- 제출 수정은 두 종류로 나뉩니다.
  - 메타데이터(제목, 알고리즘 태그 등) 수정은 덮어쓰기로 처리하며 버전을 만들지 않습니다.
  - 코드 본문 수정은 새 버전을 만듭니다.
- 제출 버전 간 diff는 추가와 삭제를 `+++`, `---` 같은 방향으로 표현합니다.
- 제출 코드 버전 목록에는 각 버전의 작성 시각과 diff를 보여줍니다.
- 코드 라인 리뷰는 제출 버전별로 분리해서 저장하고 표시합니다.
- 새 버전에서 코드 줄이 바뀐 경우 이전 버전 라인 리뷰는 새 버전 코드 라인에 매핑하지 않습니다.
- 새 버전 화면에서는 이전 버전의 라인 리뷰 존재 여부를 별도로 표시하지 않습니다.
- 제출 코드 버전은 한 방향으로만 흐르며, 예전 버전을 현재 버전으로 되돌리는 기능은 제공하지 않습니다.
- 언어 변경은 같은 제출의 수정이 아니라 별도 제출로 분리합니다.
- 같은 언어로 다시 제출해도 별도 제출로 만들 수 있습니다.
- 제출 삭제 확인 문구는 `삭제하시겠습니까?`입니다.
- 제출 삭제 시 해당 제출에 달린 댓글과 리뷰도 함께 삭제된다는 안내를 보여줍니다.
- 삭제 확인 모달의 버튼 레이블은 `삭제`로 통일합니다.
- 삭제 확인 모달의 보조 문구는 삭제 영향이 있는 대상(예: 과제 삭제)에만 표시하고, 영향이 없는 대상은 생략합니다.
- 코드 리뷰는 다른 그룹원의 제출 코드에 사용자가 직접 남기는 리뷰입니다.
- 코드 리뷰는 GitHub처럼 제출 코드의 특정 부분을 드래그해서 줄 또는 범위에 남길 수 있어야 합니다.
- 코드 리뷰는 파일 전체 또는 제출 전체에도 남길 수 있습니다.
- 같은 그룹원 모두가 코드 리뷰를 남길 수 있습니다.
- 과제 단위 댓글과 제출물 단위 댓글을 모두 지원합니다.
- 과제 단위 댓글은 일반 댓글 느낌으로, 제출물 단위 댓글은 코드 리뷰 느낌으로 사용합니다.
- 작성자가 본인의 댓글이나 리뷰를 삭제하면 실제 삭제합니다.
- 그룹장 또는 그룹 관리자가 다른 사람의 댓글이나 리뷰를 삭제하면 본문을 `삭제된 댓글입니다`로 대체합니다.
- 관리자 삭제 시 별도 삭제 사유는 입력하지 않습니다.
- 리뷰를 받은 사용자의 해결됨 표시는 제공하지 않습니다.
- 리뷰 알림은 제출 작성자에게만 보냅니다.
- 리뷰에 답글이 달리면 기존 리뷰 작성자에게도 알림을 보냅니다.
- 공지 댓글, 커뮤니티 댓글, 리뷰 답글 모두 `@` 멘션을 지원하고 멘션된 사용자에게 알림을 보냅니다.
- deadline 이후에는 미제출자도 다른 사람의 코드와 피드백을 볼 수 있습니다.
- 지각 제출자는 제출한 뒤 피드백 열람 권한을 얻습니다.
- 지각 제출을 삭제하면 해당 제출로 얻은 피드백 열람 권한도 회수합니다.
- 그룹 내부 과제 검색은 해당 그룹의 과제만 대상으로 합니다.
- 그룹 내부 과제 목록에는 필터 모달을 제공하고, 제출자(복수 선택), 해결/미해결, 출처(플랫폼), 알고리즘 태그, 제목 검색 조건을 조합해 클라이언트에서 즉시 필터링합니다.

- AI 피드백 프롬프트는 계속 테스트하면서 수정할 예정이므로 쉽게 관리할 수 있게 설계합니다.
- 캘린더는 주간 뷰를 기본으로 하고 월간 뷰로 쉽게 전환할 수 있어야 합니다.
- 같은 날짜에 과제가 여러 개 있으면 캘린더에 전부 표시합니다.
- 데스크톱에서는 날짜 셀에 과제 제목을 말줄임으로 표시하고, 모바일에서는 같은 정보를 점으로만 요약합니다. 셀 안에는 마감 시각 텍스트를 두지 않습니다.
- 날짜 셀을 누르면 해당 날짜 마감 과제 목록 모달이 열리고, 모달에서 과제를 누르면 과제 상세 페이지로 이동합니다.
- 캘린더 UX는 범용 캘린더 제품들의 패턴을 충분히 참고합니다.
- 그룹원도 캘린더에서 다른 그룹원의 제출 여부를 볼 수 있습니다.
- 그룹 규칙(`useDeadline`, `defaultDeadlineTime`, `allowLateSubmission`, `useAiFeedback`, `allowEditAfterSubmit`, `assignmentCreatorRoles`) 변경은 기존 과제에 소급 적용하지 않고 새 과제부터 적용합니다.
- 그룹 규칙이 바뀌어도 기존 제출자에게 알림을 보내지 않습니다.
- 알림 클릭 시 해당 댓글, 리뷰, 과제 위치로 이동합니다.
- 알림 대상이 삭제된 경우 이동 후 `관련 페이지가 삭제되었습니다` 메시지를 보여줍니다.
- 삭제된 과제에 대한 기존 알림을 클릭하면 `관련 페이지가 삭제되었습니다` 메시지를 보여줍니다.
- 알림 페이지는 전체 읽음 처리와 개별 읽음 처리를 모두 지원합니다.
- 알림은 사용자가 삭제하기 전까지 보관합니다.
- 알림 TTL은 두지 않습니다.
- 그룹 삭제가 발생해도 기존 알림 데이터는 삭제하지 않습니다.
- 알림 삭제 기능을 제공합니다.
- 알림 삭제는 개별 삭제와 전체 삭제를 모두 지원합니다.
- 모든 알림은 on이고, 사용자가 알림 종류별로 끌 수 없습니다.
- deadline 임박 알림은 24시간 전과 1시간 전에 보내며, 지각 제출 허용 과제에도 동일하게 보냅니다.
- 삭제된 댓글이나 리뷰에 대한 기존 알림은 남깁니다.
- 작성자가 본인의 댓글이나 리뷰를 삭제하면 실제 삭제합니다.
- 그룹장 또는 그룹 관리자가 다른 사람의 댓글이나 리뷰를 삭제하면 본문을 `삭제된 댓글입니다`로 대체합니다.
- 관리자 삭제 시 별도 삭제 사유는 입력하지 않습니다.
- 관리자 삭제로 `삭제된 댓글입니다`가 된 댓글의 답글은 그대로 유지합니다.
- 관리자 삭제된 댓글은 작성자 이름과 작성 시각을 남깁니다.
- AI 분석은 사용자별 토큰을 소모하는 방식으로 운영합니다.
- AI 분석 토큰은 성공한 분석에만 차감합니다.
- AI 토큰이 부족하면 분석 요청을 막고 `토큰이 부족합니다. 관리자에게 문의하세요.`를 보여줍니다.
- 시스템 전체 관리자가 사용자별 AI 토큰을 부여할 수 있습니다.
- 사용자별 AI 토큰 기본값은 100,000 tokens로 시작하고, 시스템 설정값으로 쉽게 변경 가능해야 합니다.
- 외부 AI API로 전송되는 데이터 범위는 구현에 필요한 최소 범위로 제한합니다.
- 사용자가 올린 코드를 AI 학습에 사용하지 않는다는 별도 정책 문구는 MVP 기획에 포함하지 않습니다.
- 제출 시 AI 분석이 허용되어 있으면 언어 재검증, 제출 코드 알고리즘 태그 추론, 개선 포인트 요약, 시간 복잡도와 공간 복잡도 추정, 잠재적인 반례 또는 엣지 케이스 제안을 한 번에 요청합니다.
- 제출 코드 AI 분석 응답은 제출 코드의 분석 결과로 저장하고, 화면에서는 저장된 분석 결과를 조회해서 보여줍니다.
- 댓글과 리뷰 알림은 즉시 각각 따로 보냅니다.
- 같은 사용자가 짧은 시간에 여러 댓글이나 리뷰를 달아도 묶음 알림 없이 각각 개별 알림으로 보냅니다.
- 댓글과 리뷰 수정 이력은 보여주지 않고 `(수정됨)`만 표시합니다.
- 제출물 전체 댓글과 드래그 기반 코드 리뷰는 같은 스레드 UI에 보여줍니다.
- 제출 코드에 대한 일반 댓글과 코드 라인 리뷰는 같은 입력 UI로 처리합니다.
- 제출 코드 diff 화면에서도 인라인 댓글을 함께 보여줍니다.
- 제출 코드 diff 화면에서는 현재 선택 버전의 변경사항과 해당 버전 인라인 댓글만 함께 보여줍니다.
- 리뷰 작성 후 수정 가능 시간 제한은 두지 않습니다.
- 제출물 전체 댓글과 드래그 기반 코드 리뷰 스레드는 오래된 순으로 정렬합니다.
- 지각 제출을 삭제해도 같은 과제에 다른 제출이 남아 있으면 피드백 열람 권한을 유지합니다.
- 전역 캘린더 탭(`/calendar`)은 내가 속한 모든 그룹 과제를 한 화면에서 보여줍니다.
- 그룹 내부 캘린더(`/groups/:groupId/calendar`)는 해당 그룹 과제만 보여줍니다.
- 그룹 캘린더 필터는 과제 상태, 제출 여부, 지각 여부, AI 분석 상태, 문제 플랫폼, 알고리즘 태그, 작성자를 지원합니다.
- 과제 검색 결과 기본 정렬은 deadline 임박순입니다.
- 과제 검색 결과 보조 정렬로 최신 과제순을 제공합니다.
- 공지와 커뮤니티는 MVP에 포함하고 같은 우선순위로 둡니다.
- 공지와 커뮤니티는 같은 게시판 모델로 설계하고 가능한 한 같은 코드를 재사용합니다.
- 공지와 커뮤니티는 파일 첨부와 이미지 첨부를 MVP에서 지원하지 않습니다.
- 공지와 커뮤니티 본문은 Markdown과 code block을 지원합니다.
- 공지 댓글을 허용합니다.
- 커뮤니티 글은 그룹원도 작성할 수 있습니다.
- 공지와 커뮤니티 글 수정 이력은 보여주지 않고 `(수정됨)`만 표시합니다.
- 커뮤니티 게시판은 카테고리를 지원합니다.
- 커뮤니티 카테고리 기본값은 `자유`입니다.
- 커뮤니티 카테고리는 일단 `자유` 하나로 시작하고 나중에 추가합니다.
- 공지 읽음 확인은 중복 조회를 제외한 사용자 수만 보여주고 읽은 사용자 목록은 보여주지 않습니다.
- 신고 기능은 고려하지 않습니다.

## 11. 다음 구현 우선순위

> 2026-05-07 시점 기준의 현재 우선순위입니다. 자세한 작업 항목과 검증 기준은 `design/checklist.md` §23 “다음 작업자 인계 노트”를 함께 봅니다.

1. **그룹 설정 화면 보강**. `/groups/:id` 설정 탭은 현재 그룹명만 변경할 수 있습니다. 신 모델(설명/최대 인원/가입 토글 4종/그룹 규칙 6항목/그룹 코드 갱신)을 모두 다루는 폼으로 확장합니다. 참고 폼은 `fe/app/groups/new/NewGroupForm.tsx`.
2. **그룹 진입 경로 단일화**. `/join-by-code`는 신 모델에서 단순화하고, 토큰 형태(코드/링크/이메일)를 자동 인식하는 단일 진입 페이지로 정리합니다. 미인증 시 `/login?next=...`로 보냅니다.
3. **가입 신청 신 모델 검증**. `joinMethods.request` off일 때 신청 생성·승인·거절 분기 테스트와, 정원 가득 상태에서의 승인 거부 테스트를 추가합니다.
4. **이메일 초대/그룹 코드 갱신 BE 테스트 보강**. 1시간 50건 초과 `RATE_LIMITED`, 7일 만료 `INVITE_EXPIRED`, 코드 갱신 후 이전 코드/링크 `404`, OWNER 외 갱신 시 `FORBIDDEN`.
5. **홈 대시보드 보강 완료**. `/`은 최근 알림, 최근 제출, 내 미제출 과제를 서버 컴포넌트에서 조합해 `해야 할 일`/`한 일` 중심 칸반 보드로 보여줍니다.
6. AI 기능(코드 리뷰·문제 분석·집단 코드 비교)의 worker 파이프라인 구현(Phase 9~10).
7. 알림 채널·SSE·deadline 임박 알림(Phase 11).
8. 캘린더·검색·공지·커뮤니티(Phase 12~13).
9. 그 외 기존 항목(과제 검색 필터/정렬, 공지 읽음 처리 등)은 해당 Phase에서 함께 닫습니다.
